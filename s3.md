## 1. S3 kya hai?

### 📖 Definition

> **S3 = Simple Storage Service**  
> *"Cloud-based storage service jo tumhe images, videos, backups, files — kuch bhi store karne deta hai — internet se kahi se bhi access karo!"*

---

### 🧠 Core Concepts — Bucket aur Object

S3 mein sirf **2 cheezein** yaad rakho:

```
S3
├── BUCKET  → Container (jaise ek badi baalti)
│   └── OBJECT → Uske andar rakhi files
│       ├── KEY   → File ka naam (e.g., photo.jpg)
│       └── VALUE → File ka content (actual data)
```

> 💡 **Analogy:** Bucket = Ek Google Drive folder | Object = Us folder mein rakhi file

---

### ⚡ Key Facts

| Feature | Detail |
|---|---|
| **Max Object Size** | **5 TB** (ek file maximum itni badi ho sakti hai) |
| **Total Storage** | Unlimited (jitna chahiye utna store karo) |
| **Pricing** | Pay per GB stored + data transfer |
| **Multi-part Upload** | 5 GB se badi files ke liye automatically use hoti hai |
| **Data Format** | Key-Value pairs as Objects |
| **Encryption** | Server-side encryption by default available |

---

### 🌍 S3 — Region Specific hai, lekin...

```
Bucket → Region specific hota hai
         (Stockholm mein banaya → Stockholm mein rahega)

BUT → Data ek region ke andar automatically
      TEENO Availability Zones mein replicate hota hai!
```

> 🛡️ **Why?** Ek data center mein power failure ya natural disaster aaye → baki 2 AZs mein data safe rahega!  
> Ye S3 khud handle karta hai — tumhe kuch extra nahi karna!

---

### 🆚 S3 vs EBS (EC2 ki hard disk)

| Feature | S3 | EBS |
|---|---|---|
| Type | Object Storage | Block Storage |
| Access | Internet se anywhere | Sirf attached EC2 se |
| Use case | Files, images, backups, websites | EC2 ka OS + app data |
| Scalability | Unlimited | Fixed size |

## 3. Static Website S3 pe Host karna

### 🌐 Concept

S3 bucket ko ek **mini website server** ki tarah treat kar sakte ho!  
HTML/CSS/Images upload karo → URL milega → Website live! 🎉

---

### 📋 Step-by-Step Process

#### Step 1: Files Upload karo

```
Apni website ki files upload karo:
  ✅ index.html
  ✅ style.css
  ✅ images (jpg/png/etc.)
```

#### Step 2: Static Website Hosting Enable karo

```
Bucket → "Properties" tab → Sabse neeche jaao
→ "Static Website Hosting" → "Edit"
→ Enable select karo
→ Index Document: index.html
→ Save Changes

✅ Ek URL mil jayega: http://bucket-name.s3-website.region.amazonaws.com
```

#### Step 3: Block Public Access Hatao

```
Bucket → "Permissions" tab
→ "Block Public Access" → "Edit"
→ Uncheck "Block all public access"
→ Save Changes
→ Confirm karo (warning aayegi — proceed karo)
```

> ⚠️ **Warning ko samjho:** Public access off karna = Data internet pe accessible hoga. Website ke liye zaroori hai.

#### Step 4: Bucket Policy Add karo

```
Bucket → "Permissions" → "Bucket Policy" → "Edit"
```

Ye JSON policy paste karo:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadGetObject",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::YOUR-BUCKET-NAME/*"
    }
  ]
}
```

> ⚡ **`/*` ka matlab:** Bucket ke andar ki **saari files** publicly readable hain

```
Save Changes → Website URL pe jaao → 🎉 Website live!
```

---

### ❓ FAQ — Block Public Access aur Policy dono kyun?

```
Block Public Access OFF karna = "Main public access allow karne ke liye READY hun"
Bucket Policy ADD karna     = "EXACTLY yahi permission de raha hun"

Dono zaroori hain:
  Block OFF → Sirf permission dene ki capability milti hai
  Policy    → Actual permission milti hai
```

## 4. Bucket Policies

### 📖 Kya hoti hain?

> **Bucket Policy = JSON format mein rules** jo batate hain:
> - **Kaun** (Principal) access kar sakta hai
> - **Kya** (Action) kar sakta hai
> - **Kaunse** (Resource) files pe

---

### 🔑 2 Important Actions

| Action | Matlab |
|---|---|
| **s3:GetObject** | Files read/download kar sako (website ke liye) |
| **s3:PutObject** | Files upload kar sako (write permission) |

---

### 🛠️ Policy banana — 2 Tarike

#### Tarika 1: Policy Generator (Easy)

```
Bucket Policy edit page → "Policy Generator" link

Step 1: Select Policy Type → S3 Bucket Policy
Step 2: Effect → Allow
Step 3: Principal → * (sabko)
Step 4: Service → Amazon S3
Step 5: Actions → GetObject
Step 6: ARN → apna bucket ARN + /* (e.g., arn:aws:s3:::my-bucket/*)
→ Add Statement → Generate Policy
→ JSON copy karo → Bucket Policy mein paste karo
```

#### Tarika 2: Documentation Examples

```
AWS Docs → S3 Bucket Policy Examples page
→ Use case ke hisaab se ready-made policy copy karo
```

> 💡 **Common Policy Examples in AWS Docs:**
> - Users ko delete karne se rokna
> - Sirf HTTPS se access allow karna
> - Specific IP se hi access allow karna

---

### 📋 Policy JSON Structure Explained

```json
{
  "Version": "2012-10-17",    // Always yahi version use karo
  "Statement": [
    {
      "Effect": "Allow",       // Allow ya Deny
      "Principal": "*",        // * = sabko, ya specific user/account
      "Action": "s3:GetObject",// Kaun sa action allow/deny karna hai
      "Resource": "arn:aws:s3:::bucket-name/*"  // Kaunsi files pe
    }
  ]
}
```

## 5. S3 Versioning

### 📖 Concept

> **Versioning = Ek hi file ke multiple versions save rehte hain**

```
index.html upload kiya (Original)  → Version: null
index.html change kiya → re-upload → Version: v1 (ID: abc123)
index.html change kiya → re-upload → Version: v2 (ID: xyz789)  ← Latest
```

> 💡 **Use case:** Galti se file delete ho gayi ya wrong version upload hua → purana version restore kar sakte ho!

---

### ✅ Versioning Enable karna

```
Bucket → "Properties" → "Bucket Versioning" → "Edit"
→ Enable → Save Changes

Note: Versioning enable karne ke BAAD upload hone wali files pe hi version IDs milenge
      Pehle ki files ka version ID = "null"
```

---

### 🔍 Versions dekhna

```
Bucket Objects page → "Show Versions" toggle ON karo

Dikhega:
  index.html  | Version: xyz789 (Latest)  | Date: Today
  index.html  | Version: abc123           | Date: Yesterday
  index.html  | Version: null             | Date: Last week (original)
```

---

### 🔄 Previous Version pe Wapas Jana

**Method 1: Current version delete karo**
```
Latest version select karo → Delete (permanently)
→ Previous version automatically "current" ban jayega
```

**Method 2: Old version download karo**
```
Old version select karo → Open/Download
→ Us file ko wapas upload karo
```

---

### 💰 Versioning ka Cost

> ⚠️ Har version alag se storage use karta hai → bill badhta hai!  
> **Best Practice:** Old versions ke liye **Lifecycle Rules** use karo (aage cover karenge)

## 6. S3 Replication

### 📖 Concept

> **Replication = Ek bucket se dusre bucket mein data automatically copy hota rehta hai**

```
Primary Bucket (Mumbai)
      ↓ Auto-replication
Replica Bucket (Mumbai ya Europe)
```

---

### 🎯 2 Types of Replication

| Type | Matlab | Use Case |
|---|---|---|
| **SRR** (Same Region Replication) | Same region mein dusra bucket | Backup within same region |
| **CRR** (Cross Region Replication) | Dusre region mein bucket | Low latency for global users + disaster recovery |

---

### 🛠️ Replication Setup — Step by Step

#### Step 1: Destination Bucket banao

```
Create Bucket → Naam do (e.g., demo-bucket-replica)
→ Region: Same ya different (apni zaroorat ke hisaab se)
→ Versioning: ENABLE karo (Replication ke liye zaroori hai!)
→ Create Bucket
```

#### Step 2: Source Bucket mein Replication Rule banao

```
Source Bucket → "Management" tab → "Replication Rules" → "Create Replication Rule"

Settings:
  Rule Name: my-replication-rule
  Status: Enabled
  Source: Apply to ALL objects (ya specific prefix/tag)
  
  Destination:
    Choose a bucket in this account
    Browse → Replica bucket select karo
  
  IAM Role: Create new role (AWS automatically banata hai)
  
  Encryption: Default rehne do
  
  Save → Done ✅
```

#### Step 3: Verify karo

```
Source bucket mein koi file upload karo
→ Wait karo (thoda time lagta hai)
→ Replica bucket refresh karo
→ File wahan bhi dikhni chahiye! ✅
```

---

### 🔍 Replication Verify karna

```
Replica bucket → "Show Versions" ON karo
→ Version IDs check karo

Source file Version ID:  3n-abc...
Replica file Version ID: 3n-abc...  ← Same! Matlab exact copy hua
```

## 7. Storage Classes

### 💰 Storage Classes kyun hain?

> Different data ke liye different zaroorat hoti hai → AWS ne alag-alag **pricing tiers** banaye hain!

```
Roz access karte ho?  → Standard (sab fast, sab expensive)
Kabhi-kabhi?          → Standard-IA (thoda sasta)
Sirf backup ke liye?  → Glacier (bahut sasta)
10 saal ke liye rakhna? → Glacier Deep Archive (sabse sasta)
```

---

### 📊 Storage Classes Comparison Table

| Storage Class | Use Case | Cost | Retrieval | Min Duration |
|---|---|---|---|---|
| **S3 Standard** | Frequently accessed data, websites | Highest | Instant | None |
| **S3 Intelligent-Tiering** | Unknown/changing access patterns | Slightly higher | Instant | None |
| **S3 Standard-IA** | Infrequently accessed, still fast retrieval | Lower | Instant | 30 days |
| **S3 One Zone-IA** | Non-critical, infrequent — sirf 1 AZ | Cheapest among IA | Instant | 30 days |
| **S3 Glacier Instant** | Archive but fast retrieval needed | Low | Instant | 90 days |
| **S3 Glacier Flexible** | Long-term archive, hours retrieval ok | Very Low | Minutes-Hours | 90 days |
| **S3 Glacier Deep Archive** | 10+ saal archive, compliance | Cheapest overall | Hours | 180 days |
| **S3 Outposts** | On-premises AWS Outpost storage | Variable | Instant | N/A |

---

### 🎯 Use Case → Storage Class Mapping

| Scenario | Recommended Class |
|---|---|
| Static website files (daily access) | **S3 Standard** |
| User profile images (frequent access) | **S3 Standard** |
| Log files (monthly analysis) | **S3 Standard-IA** |
| Old database backups | **S3 Glacier Flexible** |
| Legal/compliance documents (10 years) | **S3 Glacier Deep Archive** |
| App with unpredictable usage | **S3 Intelligent-Tiering** |
| Test data — single AZ ok | **S3 One Zone-IA** |

---

### 🔧 Storage Class kaise change karein?

```
Bucket → File pe click karo → Properties → "Storage Class" → Edit
→ Dropdown mein se class choose karo → Save
```

> 💡 **Best practice:** Manually change mat karo — **Lifecycle Rules** use karo (automated!)

## 8. Lifecycle Rules

### 📖 Concept

> **Lifecycle Rules = Automatic rules jo data ko ek storage class se dusri mein move karte hain ya delete karte hain — time ke basis pe**

```
Day 0:   File upload → S3 Standard (expensive, fast)
Day 30:  Auto-move  → S3 Standard-IA (cheaper)
Day 60:  Auto-move  → S3 One Zone-IA (even cheaper)
Day 90:  Auto-move  → S3 Glacier (very cheap — archive)
Day 365: Auto-delete → Gone! (agar zaroorat nahi)
```

> 💡 **Why?** Paisa bachao + manually sab nahi karna padta!

---

### 🛠️ Lifecycle Rule banana

```
Bucket → "Management" tab → "Lifecycle Rules" → "Create Lifecycle Rule"

Step 1: Rule Name → my-lifecycle-rule
Step 2: Scope → All objects ya specific prefix/tag

Step 3: Actions:
  ✅ Transition current versions of objects between storage classes
  ✅ Expire current versions of objects (auto-delete)

Step 4: Transition configure karo:
  Standard → Standard-IA: after 30 days
  Standard-IA → One Zone-IA: after 60 days
  One Zone-IA → Glacier: after 90 days

Step 5: Expiration (optional):
  Current version expire after: 365 days (auto-delete)

→ Save → Done ✅
```

---

### 📊 Lifecycle Rule Visual

```
Timeline:
  ├── Day 0    → Upload → S3 Standard
  ├── Day 30   → Auto   → S3 Standard-IA
  ├── Day 60   → Auto   → S3 One Zone-IA
  ├── Day 90   → Auto   → S3 Glacier
  └── Day 365  → Auto   → DELETED ✅

Cost trend: ↓ ↓ ↓ ↓ (time ke saath cost kam hoti jaati hai)
```

## 9. Snow Family

### 📖 Problem Statement

```
Tumhare paas 10 TB ka data hai → S3 pe migrate karna hai

Internet upload speed: 100 Mbps
10 TB = 10,000 GB = 80,000 Gbits

Time = 80,000 / 0.1 = 800,000 seconds = ~9 DAYS! 😱

Solution? → AWS Snow Family!
```

### 🌨️ Snow Family kya hai?

> **Physical devices jo AWS bhejta hai tumhare paas → Data copy karo → Device wapas bhejo → AWS S3 mein upload kar deta hai!**

```
AWS → Physical Device Ship karta hai
   ↓
Tumhara Data → Device mein copy karo
   ↓
Device wapas ship karo AWS ko
   ↓
AWS → S3 mein data upload kar deta hai ✅
```

---

### 📦 3 Types of Snow Devices

| Device | Size | Use Case | Analogy |
|---|---|---|---|
| **Snow Cone** | Few TBs | Small data migration, portable | Pen drive / portable HDD |
| **Snowball Edge** | Petabytes | Large data + Edge Computing | Server-size suitcase |
| **Snow Mobile** | Exabytes | Entire data center migration! | 🚛 Literally a truck! |

---

### 🔄 Snow Family ke 2 Use Cases

```
1. IMPORT → On-premises se S3 mein data laana
   (Office se → AWS cloud mein)

2. EXPORT → S3 se data wapas physical device mein laana
   (AWS cloud se → tumhare paas)
```

---

### 📋 Order kaise karte hain?

```
AWS Console → Search "Snow Family"
→ "Order an AWS Snow Family device"
→ Job Type: Import into S3 / Export from S3
→ Device type choose karo (Snowcone/Snowball/Snowmobile)
→ Storage capacity choose karo (1TB/210TB/etc.)
→ Shipping address do
→ IAM Role assign karo
→ Order! → Device deliver hoga → Data copy karo → Wapas bhejo
```

## 10. S3 Storage Gateway

### 📖 Concept

> **Storage Gateway = Hybrid Cloud Storage Bridge**  
> On-premises server (apna data center) ko S3 se connect karne ka tarika

```
Your On-Premises Server/App
          ↓
   [Storage Gateway]  ← Bridge ka kaam karta hai
          ↓
      AWS S3 Bucket
```

---

### 🎯 Use Case

```
Scenario: 
  - Tumhari application apne data center pe run hoti hai
  - Kuch specific data S3 pe store karna chahte ho
  - Ya S3 ka data locally access karna chahte ho

Solution: Storage Gateway!
```

---

### 3 Types of Storage Gateway

| Type | Matlab |
|---|---|
| **File Gateway** | Files ko S3 Objects ki tarah store/access karo (NFS/SMB protocol) |
| **Volume Gateway** | Block storage — iSCSI volumes as S3 backups |
| **Tape Gateway** | Virtual tape library — legacy backup systems ke liye |

## 11. Encryption

### 🔐 S3 Encryption

> By default, S3 **server-side encryption** enable hoti hai!

```
File upload karo → AWS automatically encrypt karta hai → S3 mein store hoti hai
File access karo → AWS automatically decrypt karta hai → Tumhe milti hai
```

---

### Encryption check karna

```
Bucket → Koi file click karo → Properties tab
→ "Server-side encryption" → Enabled dikhega ✅
```

---

### Encryption ke Types

| Type | Matlab |
|---|---|
| **SSE-S3** | AWS ke keys se encrypt (default) |
| **SSE-KMS** | AWS Key Management Service ki keys se |
| **SSE-C** | Tumhari apni custom keys se |
| **Client-side** | Upload se pehle hi encrypt karo |

> Learning phase mein **SSE-S3 (default)** kafi hai!

### ⚡ Common Mistakes to Avoid

```
❌ Bucket naam globally unique nahi rakha → Error aayega
❌ Block Public Access OFF nahi kiya → Website access nahi hogi
❌ Bucket Policy mein /* (slash star) nahi lagaya → Files access nahi hongi
❌ Replication ke liye destination bucket mein versioning enable nahi ki
❌ Versioning enable karne ke baad forget kar diya → Bill badh sakta hai
❌ Testing ke baad buckets delete nahi kiye → Bill aayega (empty karo pehle, phir delete)
```

```

        → Bucket naam type karo confirm karne ke liye → Delete ✅
```
